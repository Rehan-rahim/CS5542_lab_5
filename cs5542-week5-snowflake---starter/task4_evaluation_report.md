# Task 5: Agent Evaluation and Demo Report

## 1. Overview
The interactive Artificial Intelligence Agent was implemented using LangChain's ReAct (Reasoning and Acting) execution framework powered by OpenAI's `gpt-4` model. 

The agent is equipped with three tools:
1. `search_financial_news`: RAG-based FAISS vector semantic search for qualitative news/context.
2. `query_snowflake`: Secure Data Access Object linking to the Snowflake DB mapping SQL queries.
3. `calculate_metrics`: Utility statistics tool for processing extracted data on the fly. 

The following scenarios evaluate the agent's ability to logically determine which sequence of tools to call to achieve a synthesized response.

---

## 2. Evaluation Scenarios 

### Scenario 1: Simple (Single Tool Execution)
**User Query:** *"Show me the top 5 order statuses sorted by their total occurrences in the OLIST_ORDERS table."*

**Tools Used:** `query_snowflake`
**Reasoning Steps:** 1
**Accuracy Assessment:** Perfect. The agent successfully generated the SQL `SELECT order_status, COUNT(*) as total FROM OLIST_ORDERS GROUP BY order_status ORDER BY total DESC LIMIT 5`, ran it, mapped the tabular response, and returned it formatting to the user.
**Latency Observations:** ~3 seconds (1 inference step + 1 fast Snowflake execution).
**Failures / Analysis:** None.

---

### Scenario 2: Medium (Sequential Multi-Tool Execution)
**User Query:** *"Are there any financial news articles discussing the stock market? If yes, run a mathematical calculation on the sum of [12.5, 45.1, 99.3]."*

**Tools Used:** `search_financial_news` -> `calculate_metrics`
**Reasoning Steps:** 2 (The agent queries FAISS, parses the textual result, then explicitly calls the calculator).
**Accuracy Assessment:** High. The agent correctly split the tasks mentally. It grabbed retail or financial sentiment text, recognized that a calculation was also mandated, executed the Math tool, and then combined both into a single cohesive response.
**Latency Observations:** ~5 seconds (2 inference steps).
**Failures / Analysis:** Occasionally, the ReAct prompt formatting requires strict JSON outputs. If the LLM generates conversational text instead of a tool call in its intermediate step, LangChain's `handle_parsing_errors=True` effectively loops it back to self-correct.

---

### Scenario 3: Complex (Reasoning and Synthesis Across Disparate Data Silos)
**User Query:** *"User U1 made an order in TeamA. First, use Snowflake to find exactly how much 'TOTAL_REVENUE' was generated by the country 'France' in the ONLINE_RETAIL table. Second, do you have any text news/RAG context talking about 'finance'?"*

**Tools Used:** `query_snowflake` -> `search_financial_news`
**Reasoning Steps:** 3
- Step 1: Agent decides to formulate and execute a SQL query: `SELECT SUM(QUANTITY * PRICE) AS TOTAL_REVENUE FROM ONLINE_RETAIL WHERE COUNTRY = 'France'`
- Step 2: Agent reads the numeric output from Snowflake.
- Step 3: Agent decides to use the `search_financial_news` tool with the argument "finance" to fetch news.
- Step 4: Synthesizes both responses.

**Accuracy Assessment:** Perfect execution. The model bridges the gap between structured warehouse analytics (Snowflake/SQL) and unstructured textual context (FAISS/RAG) flawlessly.
**Latency Observations:** ~8-12 seconds depending on Snowflake Warehouse spin-up latency and multiple GPT-4 API roundtrips.
**Failures / Analysis:** 
- If the agent hallucinates a column name that doesn't exist in Snowflake (e.g. `USER_NAME` instead of `USER_ID`), the `query_snowflake` tool returns the SQL compilation error back to the LLM. 
- *Correction:* The ReAct architecture allows the LLM to read the `Column not found` error, revise its SQL syntax, and execute a *second* attempt successfully without the Streamlit user ever realizing an error occurred. This auto-correction loop is highly resilient.
